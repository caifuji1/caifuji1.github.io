RDB与AOF

持久化方式 RDB AOF
占用存储空间 小（数据级：压缩） 大（指令级：重写）
存储速度 慢 快
回复速度 快 慢
数据安全性 会丢失数据 依据策略决定
资源消耗 高/重量级 低/轻量级
启动优先级 低 高

事务

multi

exec

生成一个队列把操作放入然后执行

可以用discard来取消事务

如果在定义事务过程中，命令个数输入错误，那么整个事务将不会执行

如果在执行中出现错误，那错误的命令将不会执行

锁

监视锁watch如果在事务执行前key发生了变化，终止事务执行，不能在事务中写。

unwatch取消所有监视

分布式锁。互斥访问类似

数据删除策略

设置过期时间的数据都会在内存中开一个expires空间进行存放（放的是数据地址和过期时间）

1、定时删除，一个计时器过期了就删除

2、惰性删除，下次访问数据的时候进行删除

3、定期删除，

Redis启动服务器初始化时，读取配置server.hz的值默认为10

每秒钟执行server.hz次serverCron（），在databasesCron（）进行db的轮询，activeExpireCycle（）对每个expires进行逐一检测，每次执行250ms/server.hz

随机挑选W个key检测

1.如果key超时就删除

2.如果一轮中的key的数量>W*25%，循环该过程

3.key<=W*25%，检查下一个expires

W取值可以进行配置ACTIVE_EXPIRE_CYCLE_LOOKUPS_PER_LOOP

activeExpireCycle()中有个current_db用于记录进入哪个expires

删除策略对比

定时删除 节约内存，无占用 不分时段占用CPU资源，频度高 拿时间换空间
惰性删除 内存占用严重 延时执行，CPU利用率高 拿空间换时间
定期删除 内存定期随机清理 每秒花费固定时间的CPU资源维护内存 随机抽查，重点抽查

逐出算法

redis在每个命令执行前都会调用freeMemoryifNeeded（）方法检测内存是否充足，Redis会临时淘汰一些数据，这种策略就是逐出算法

不一定能清理出足够的内存，他会反复执行，当对所有数据都尝试后还是不行那么就会报出异常

maxmemory配置可用的最大内存

maxmemory-samples每次选取删除数据的个数

maxmemory-policy删除策略

检测易失数据（LRU：Least Recently Used，LFU：LeastFrequently Used）

server.db[i].expires

volatile-lru：挑选最近使用最久未使用

volatile-lru：使用次数最少的

volatile-ttl：要过期的数据

volatile-random：任意选择数据淘汰

检测全库数据

server.db[i],dict

allkeys-lru：挑选最近使用最久未使用

allkeys-lru：使用次数最少的

allkeys-random：任意选择数据淘汰

放弃数据驱逐

no-enviction（驱逐）：禁止驱逐数据（redis4.0中没人策略），会引发错误OOM（Out Of Memory ）

Redis集群

主从复制

读写分离：master写、slave读，提高服务器的读写负载能力

负载均衡：提高Redis服务器并发量和数据吞吐量

故障恢复：当master出现问题时，由slave提供服务，实现快速故障恢复

数据冗余：数据热备份，是持久化之外的一种数据冗余方式

高可用的基础，基于主从复制，构建哨兵模式与集群，实现Redis高可用方案

主从复制过程3个阶段

​ 1.建立连接阶段

​ 2.数据同步阶段

​ 3.命令传播阶段

建立连接阶段

slaveof 主机 端口

或者在启动服务的时候redis-server –slaveof 主机 端口 也能够连接

主流方式走配置文件，加入slaveof 127.0.0.1 6379

info可以查看到连接信息

slaveof no one 断开连接

数据同步阶段

slave发送指令psync2

master

在执行全量复制的时候，如果周期时间过长，部分复制的复制缓冲区回造成数据丢失。然后就会执行第二次全量复制。如此往复就会造成死循环。

解决修改repl-backlog-size 1mb 这个是默认大小

slave

避免slave进行全量复制、部分复制时服务器响应阻塞或不同步，建议关闭期间的对外服务功能slave-serve-stale-data yes|no

多个slave同时对master请求数据同步就会造成RDB文件增多，对带宽造成冲击，因此要错峰进行

slave过多时，建议调整拓扑结构。注意层级深度越高，顶层与底层之间数据同步 延迟较大，数据一致性较差，应谨慎选择

命令传播阶段

网络闪断闪连 忽略

短时间网络中断 部分复制

上齿间网络中断 全量复制

服务器运行ID（runid）：服务器运行的身份识别码（40位字符组成），用于服务器间进行传输识别身份

复制积压缓冲区中有偏移量和字节值。他是一个队列

心跳机制

命令传播阶段信息交互就是用心跳机制进行维护，实现双方连接保持在线

master心跳：

​ 指令：PING

​ 周期：由repl-ping-slave-period决定，默认10s

​ 作用：判断slave是否在线

​ 查询：INFO replication 获取slave最后一次连接时间间隔，lag是0或1都视为正常

slave心跳任务：

​ 指令：REPLCONF ACK{offset}

​ 周期：1s

​ 作用1：汇报slave自己的复制偏移量，获取最新的数据变更指令

​ 作用2：判断master是否在线

注意事项

当slave多数掉线，或延迟过高时，master为保障数据的稳定，将拒绝数据的同步操作

min-slaves-to-write 2

min-slaves-max-lag 8

当slave数量少于2个，或者延迟大于等于8时，强制关闭master写功能，停止数据同步

频繁的全量复制问题

master内部创建master_replid变量和runid相同策略生成，41位长度发给所有的slave

在master关闭时执行shutdown save，进行RDB持久化，将runid和offset保存进去

redis-check-rdb命令查看

重启后加载RDB文件，恢复数据

作用：重启后恢复该值，让slave认为还是之前的master

频繁网络中断问题

master每1秒调用复制定时函数replicationCron（），对比slave发现长时间没有相应

master各种资源（输入缓冲区、带宽、连接等）被严重占用

解决：设置超时时间，确认是否释放slave

repl-timeout 默认60s

问题：slave与master连接断开

问题原因：master发送ping指定频度较低

​ master设置超时时间较短

​ ping指定在网络中存在丢包

解决：提高ping指令发送的频度

repl-ping-slave-period 超时时间repl-time的时间至少是ping指令频度的5-10倍，否则slave很容易判定超时

数据不同步问题

网络问题，发送数据有延迟

解决：优化主从之间的网络环境，通常在同一机房部署，如果使用云服务器要注意

监控主从节点延迟（通过offset）判断，如果过大暂时屏蔽程序对slave的数据访问

slave-serve-stale-data yes|no

开启后仅相应info、slaveof等少数命令（慎用，除非对数据一致性非常高）

哨兵sentinel

是一个分布式系统

监控

不断检查master和slave是否正常运行

master存活检测、master与slave与运行情况检测

通知

当被监控的服务器出现问题是，向其他（哨兵，客户端）发送通知

自动故障转移

断开master与slave连接，选取一个slave 作为master，重新连接，并告知客户端新的服务器地址

哨兵也是一台Redis服务器，只是不提供数据服务，通常哨兵配置数量为单数

sentinel向master发起，，master断了就会flags：SRI_S_DOWN，然后在sentinel中共享，之后其他的sentinel也会向master发起，确认挂了flags：SRI_O_DOWN（只要半数以上确认挂了，就挂了）

flags：SRI_S_DOWN这个主观下线

flags：SRI_O_DOWN客观下线

故障转移阶段，选出哪个sentinel来处理，发送的信息有

挂的ip

挂的端口

竞选次数

自己的runid

这里面有个投票机制，谁先收到投给谁，如果没投出，竞选次数加一再次

然后选出的sentinel在slave中选出master

在线的

响应快的

与原master断开时间最短的

优先原则

​ 优先级

​ offset

​ runid

然后向新的master发送slaveof no one，向其他slave发送slaveof 新的masterIP端口

之后原来的master启动后就变为slave去连接现在的master

Redis集群

多主多从

减小服务器访问压力，分散单台服务器存储压力，降低服务器宕机带来的业务灾难

存储空间分割为16384份，用CRC16

集群内部通讯设置

各个数据库相互通信，保存各个库槽的编号数据

一次命中，直接返回

一次未命中，告知具体位置，然后客户端直接去找具体位置。

配置

cluster-enable yes 设置加入cluster

cluster-config-file nodes-6379.conf cluster配置文件名

cluster-node-timeout 100000 超时时间

cluster-migration-barrier 1 master连接的slave最小数量

连客户端的时候redis-cli -c

slave下线后，master会标记一下这个从不能用了，不影响其他

命令

cluster node 查看集群节点信息

cluster replicate 进入一个从节点redis，切换其主节点

cluster meet ip:port 发现一个性节点，新增主节点

cluster forget id 忽略一个没有solt的节点

cluster failover 手动故障转移
